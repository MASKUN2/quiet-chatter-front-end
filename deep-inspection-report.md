# 🔍 QuietChatter Frontend - 심층 코드 인스펙션 리포트 (Deep-Dive)

본 리포트는 초기 분석에서 발견되지 않았던 애플리케이션의 **구조적 성능 저하 요인, 메모리 누수, 잠재적 보안 결함, 그리고 React 안티패턴**에 초점을 맞춘 심층 검사(Deep-dive Inspection) 결과입니다.

---

## 1. ⚡ 성능 개선 및 버그 방지 (Performance & Robustness)

### 1.1. 코드 스플리팅(Code Splitting) 부재로 인한 초기 로딩 지연
- **위치:** `src/App.tsx`
- **문제점:** 애플리케이션의 모든 페이지 컴포넌트(`Home`, `BookSearch`, `BookDetail`, `MyPage` 등)가 최상단에서 정적(Static)으로 임포트되어 라우팅되고 있습니다. 애플리케이션 규모가 커질수록 JavaScript 번들 사이즈가 거대해져, 사용자가 처음 진입할 때 무거운 파일을 한 번에 다운로드해야 하므로 **초기 렌더링 시간(TTI)**이 크게 지연됩니다.
- **개선 방안:** `React.lazy`와 `<Suspense>`를 도입하여 라우트 기반 코드 스플리팅을 적용해야 합니다. 사용자가 특정 페이지에 접근할 때만 해당 페이지의 스크립트를 비동기적으로 불러오도록 구조를 변경해야 합니다.

### 1.2. 언마운트 시 타이머(Timer) 누수
- **위치:** `src/pages/NaverCallback.tsx`
- **문제점:** 네이버 로그인 콜백 후 페이지 리다이렉트를 위해 `setTimeout(() => navigate(...), 2000)`과 같은 로직이 다수 사용되고 있습니다. 만약 이 타이머가 완료되기 전에 사용자가 뒤로 가기를 누르거나 컴포넌트가 언마운트되면, 타이머 콜백이 백그라운드에서 실행되어 **메모리 누수**가 발생하거나 의도치 않은 라우팅 에러가 발생할 수 있습니다.
- **개선 방안:** `useEffect` 내부에서 타임아웃의 식별자(ID)를 변수에 할당하고, cleanup 함수(return 구문)에서 `clearTimeout`을 호출하여 컴포넌트 종료 시 펜딩된 타이머를 취소해야 합니다.

### 1.3. 스켈레톤 UI의 배열 Index as Key 안티패턴
- **위치:** `src/components/home/RecommendedTalks.tsx`
- **문제점:** 로딩 중 뼈대를 보여주는 스켈레톤 리스트에서 `{Array.from(new Array(6)).map((_, index) => (<ListItem key={index} ...>))}`처럼 배열의 인덱스를 직접 컴포넌트의 `key`로 사용하고 있습니다. 정적인 UI에서는 동작에 치명적인 문제는 없으나, 이는 React의 리렌더링 최적화를 방해하는 대표적인 안티패턴입니다.
- **개선 방안:** 고유한 값이나 식별 가능 문구(`key={\`skeleton-talk-\${index}\`}`)로 `key` 값을 개선하는 것을 권장합니다.

## 2. 🛡️ 잠재적 보안 취약점 (Security Vulnerabilities)

### 2.1. 개발 환경 변수(프록시 타겟) 하드코딩
- **위치:** `vite.config.ts`
- **문제점:** 프록시 서버 설정 내 타겟 URL(`https://dev-api.quiet-chatter.com`)이 소스 코드에 직접 기입되어 있습니다. 프록시 타겟이나 인증 관련 서버 주소가 하드코딩되면 인프라 변경 시 소스 코드를 재수정해야 하고, 프로덕션 환경의 정보가 깃(Git) 히스토리에 노출될 위험이 있습니다.
- **개선 방안:** Vite의 환경 변수 주입 기능(`loadEnv` 또는 `process.env.VITE_API_URL`)을 활용하여 `.env` 파일로 중요 서버 주소를 분리해야 합니다.

## 3. 🛠️ 사용자 경험 및 상태 처리(UX / Error Handling)

### 3.1. API 호출 실패 시 사용자 피드백 누락 (Silent Failure)
- **위치:** `src/hooks/useBookDetail.ts` (`loadBook`, `loadTalks` 등)
- **문제점:** 데이터를 초기 로드하는 비동기 함수들이 `try-catch` 블록으로 감싸져 있으나, 실패 시 `console.error(error)`만 남길 뿐 상태(`error`)를 설정하거나 화면에 토스트(Toast) 메시지를 띄우지 않습니다.
- **영향:** 네트워크 단절 또는 백엔드 에러 시 사용자는 영원히 스피너(Loading)만 보게 되거나 텅 빈 화면을 보게 됩니다. 에러가 발생했음을 인지할 수 없는 **Silent Failure** 상태가 됩니다.
- **개선 방안:** 폼 제출 외에도 데이터 페칭 시 실패할 경우, `useToast`를 통해 에러를 알리거나 상태(`hasError`)를 업데이트하여 UI에서 대체 화면(Fallback View)을 보여주어야 합니다.
